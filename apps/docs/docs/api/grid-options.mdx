# GridOptions

`GridOptions<TData>` is exported from `@onegrid/core` and contains shared grid configuration for
core, DOM, React, and Vue. DOM mount targets are intentionally kept in `@onegrid/dom` as
`DomGridOptions<TData>` so core remains browser-independent.

Column UI options are shared across DOM, React, and Vue:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  columnUi: {
    resize: true,
    autoSize: true,
    reorder: true,
    menu: true,
    toolPanel: true
  }
};
```

`defaultColumnDef` applies reusable defaults to every leaf column before explicit column options.
`columnTypes` names reusable column profiles that a column can reference with `type`:

```ts
const options: GridOptions<OrderRow> = {
  columns: [
    { field: "id", headerName: "ID", pinned: "left" },
    { field: "amount", headerName: "Amount", type: "money", width: 180 },
    { field: "requestedAt", headerName: "Requested", type: "dateOnly" }
  ],
  defaultColumnDef: {
    minWidth: 96,
    resizable: true,
    sortable: true,
    filter: "text"
  },
  columnTypes: {
    money: {
      type: "number",
      width: 140,
      filter: "number",
      editor: "number",
      formatter: ({ value }) => Number(value).toLocaleString("en-US")
    },
    dateOnly: {
      type: "date",
      width: 132,
      filter: "date"
    }
  }
};
```

Merge order is `defaultColumnDef`, then any type defaults, then the explicit column definition.
Column types can inherit from built-in or named types by setting their own `type`.

`columnState` can be supplied by renderers or persisted application state when column width,
visibility, order, pinning, or column group open state should be restored. The runtime API uses the
same shape through `api.getColumnState()`, `api.setColumnState()`, `api.applyColumnState()`,
`api.setColumnGroupOpen()`, `api.toggleColumnGroup()`, and `api.resetColumnState()`. Prefer
`applyColumnState()` for user preference restore because it validates missing column and group ids
and can apply partial state without replacing unrelated column state.

`initialState` restores mutable grid state without mixing it into static options. It is the preferred
contract for saved user views, route restore, and wrapper parity:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  initialState: {
    columnState: {
      columns: { amount: { width: 160, hidden: false } },
      groups: { workflow: { open: true } }
    },
    sortModel: [{ columnId: "amount", field: "amount", direction: "desc" }],
    filterModel: { quickText: "approved" },
    groupModel: { fields: ["region"], expandedKeys: ["group:region=Capital"] },
    rowModelState: {
      rowModel: "server",
      page: 1,
      expandedGroupKeys: ["group:region=Capital"]
    },
    selection: { mode: "row", rowKeys: ["ORD-001"], cells: [], ranges: [] },
    pagination: { page: 2, pageSize: 50 },
    scroll: { top: 360, left: 120 },
    locale: "ko-KR"
  }
};
```

`initialState` is intentionally limited to runtime UI state. It does not replace `columns`, `data`,
`dataSource`, renderer definitions, or security options. Missing fields are left at normal defaults.
`rowModelState` restores remote/tree row-model cursors and expansion metadata, but it does not
embed row data or cache contents.

`rowKey` must resolve to a stable row id for selection, editing, clipboard, merge layout, and state
restore. When `rowKey` is configured, duplicate ids are treated as configuration/data errors by
default:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  duplicateRowKeyPolicy: "error"
};
```

Use `duplicateRowKeyPolicy: "suffix"` only for legacy datasets that cannot be cleaned immediately.
Suffix mode keeps the first row id unchanged and assigns later duplicates synthetic ids such as
`ORD-1__2`; it should not be used for audited transaction paths.

`dataSourceOptions` standardizes remote request retry and status behavior for infinite, server,
viewport, and lazy tree row models:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  rowModel: "server",
  dataSource,
  dataSourceOptions: {
    retry: {
      attempts: 3,
      delayMs: 100,
      maxDelayMs: 1000,
      backoff: "exponential",
      retry: ({ error }) => error.statusCode !== 400
    }
  }
};
```

Failures are normalized to `DataSourceError`. Row models expose a `DataSourceStatusSnapshot`, and
DOM/React/Vue receive the shared `dataRequested`, `dataLoaded`, and `error` events.

`beforeEvents` provides cancellable hooks before grid mutations. Use it for business rules that
must block a user action before the grid changes visible state:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  beforeEvents: {
    beforeColumnStateChange(event) {
      if (event.event.columnState.columns?.orderId?.hidden) {
        event.preventDefault("Order ID cannot be hidden");
      }
    },
    beforePageChange(event) {
      if (hasUnsavedChanges()) {
        event.preventDefault("Save edits before paging");
      }
    }
  }
};
```

The same hooks can be attached at runtime through `GridApi.onBefore()`. React and Vue expose the
same contract through `beforeEvents` plus framework event props/emits.

Client row model inputs are supplied through the existing row options:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  rowModel: "client",
  filtering: { model: filterModel },
  sorting: { model: sortModel },
  grouping: { model: groupModel },
  aggregation: { model: aggregateModel }
};
```

When `serverOnly` is enabled on a row option, the DOM renderer does not apply that client-side model.

Sorting options control the shared `SortModel[]` used by client, infinite, server, and viewport row
models:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowModel: "client",
  sorting: {
    enabled: true,
    multiSort: true,
    sortOrder: ["asc", "desc", null],
    model: [{ field: "amount", direction: "desc" }]
  }
};
```

`multiSort` enables Shift-click additive sorting in the DOM renderer. `serverOnly` prevents client
row sorting while preserving the model for data-source requests.

Filtering options control the shared `FilterModel` used by quick filter, column menu filters, and
row model requests:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowModel: "client",
  filtering: {
    enabled: true,
    quickFilter: true,
    model: {
      quickText: "office",
      conditions: [{ field: "status", kind: "set", operator: "in", value: ["Approved"] }]
    }
  }
};
```

`serverOnly` prevents client filtering while preserving the model for `DataSource.getRows`.

Grouping options control the shared `GroupModel` used by client and server row models:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  rowModel: "client",
  grouping: {
    enabled: true,
    footer: "bottom",
    model: {
      fields: ["region"],
      expandedKeys: ["group:region=Seoul"]
    }
  },
  aggregation: { model: aggregateModel }
};
```

`footer: "bottom"` renders group footer rows with aggregate values. `api.expandGroup()`,
`api.collapseGroup()`, and `api.toggleGroup()` update the same group model at runtime. In server
row models, expanding a group updates `server.groupKeys` for the next data-source request.

Editing options control lifecycle policy while column definitions own individual editor kinds,
parsers, validators, and value setters:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  rowModel: "client",
  editing: {
    enabled: true,
    startMode: "doubleClick",
    commitMode: "batch",
    undoRedo: { enabled: true, limit: 100 },
    readOnly: false,
    keyboard: {
      startOnEnter: true,
      commitOnEnter: true,
      moveOnTab: true,
      commitOnTab: true,
      cancelOnEscape: true,
      clearOnBackspace: true
    },
    blurAction: "cancel",
    validateOnCommit: true
  }
};
```

The DOM renderer starts editing from Enter, F2, printable keys, pointer edit policy, or
`GridApi.startEdit()`. `startMode` controls pointer-started editing: `doubleClick` is the default for
regular cells, `singleClick` starts editing from a primary click, and `manual` suppresses click and
double-click editing while preserving keyboard/API entry points. Checkbox editors default to
`singleClick` unless the column or grid option overrides the policy. `blurAction` controls whether
focus loss commits or cancels the editor.
`keyboard` makes editing keys explicit. The default enterprise policy starts editing from Enter,
commits single-line editors from Enter, commits and moves from Tab, cancels from Escape, and clears
then starts editing from Backspace. `moveOnTab` controls focused-cell Tab navigation and the focus
move after an editor Tab commit.
`commitMode: "batch"` stores accepted values in pending edit history until
`GridApi.commitPendingEdits()` finalizes them; `GridApi.cancelPendingEdits()` restores the previous
values.
`undoRedo` controls the cell edit undo/redo stack. It is enabled by default with a bounded history
limit, can be disabled with `undoRedo: false`, and is exposed through `GridApi.undoEdit()`,
`redoEdit()`, `getEditHistoryState()`, and `clearEditHistory()`.
`readOnly: true` keeps the same editor lifecycle but emits `cellEditRequested` instead of mutating
or staging row data. Use it for externally controlled data stores and then feed accepted
`event.nextRow` values back through `setData`, `updateRows`, React props, or Vue props.
`commitOnBlur` remains supported as a legacy boolean alias, but `blurAction` is preferred for new
code. React and Vue pass the same `editing` option through to the DOM renderer.

Selection options control the shared `GridSelectionState` used by pointer, checkbox, keyboard, and
API-driven selection:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  rowModel: "client",
  selection: {
    mode: "range",
    multiple: true,
    checkbox: true,
    selectAll: "server",
    serverSelectionToken: "orders"
  }
};
```

`selectAll: "visible"` selects only rendered row keys. `selectAll: "server"` creates a token from
the current row model, filter model, sort model, and snapshot version without loading the whole
dataset into the client.

Clipboard options enable browser text/plain copy/paste without exposing HTML from renderers:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  selection: { mode: "range", multiple: true },
  editing: { enabled: true, validateOnCommit: true },
  clipboard: {
    enabled: true,
    includeHeaders: true,
    preserveMerge: true
  }
};
```

`includeHeaders` controls the default header row for copy operations. `preserveMerge` writes merge
anchor values once and blank cells for covered coordinates.

Export and import options configure file conversion defaults for `GridApi.exportData()` and
`GridApi.importData()`:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  export: {
    format: "xlsx",
    includeHeaders: true,
    includeHeaderMerges: true,
    includeCellMerges: true,
    preserveVisualLayout: true,
    filename: "orders.xlsx"
  },
  import: {
    format: "csv",
    mode: "replace",
    hasHeaders: true,
    headerRowCount: 1,
    columns: ["id", "region", "amount"],
    parseRow: (record) => ({
      id: String(record.id),
      region: String(record.region),
      amount: Number(record.amount)
    })
  }
};
```

CSV uses blank covered cells for visual merge preservation. XLSX and print HTML also preserve merge
ranges. XLSX files include frozen exported header rows, autofilter metadata, numeric cells, borders,
fills, and print setup. PDF export is a lightweight paginated text-table document with repeated
headers and horizontal column pages, while print HTML is the richer layout path for browser printing.

Import uses `mode: "replace"` by default so a file import clears the current client rows before
rendering imported rows. `mode: "append"` is opt-in for workflows that intentionally add imported
rows beneath the current dataset. `headerRowCount` controls how many visual header rows are skipped
before parsing body records, which is required when importing a OneGrid XLSX export with merged
multi-row headers. XLSX import reads shared strings, inline strings, cached formula values, formula
text without a cached value, and date-formatted numeric cells before `parseRow` receives the record.

Header merge rules are validated before rendering. A rule must reference leaf data column ids, row
presentation rules must be contiguous and non-overlapping, label presentation rules must map to one
existing header cell, and rule ids must be unique. Invalid rules throw during grid creation so saved
views and wrapper examples fail early instead of rendering a broken header.

Localization options select the active locale for renderer chrome and formatter context helpers:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  locale: "ko-KR"
};
```

`CellContext` includes `locale`, `formatNumber()`, and `formatDate()` so column formatters can use
the same locale as footer, pagination, overlays, and ARIA live-region messages. Built-in locales
are `en-US` and `ko-KR`; custom definitions are registered through `registerLocale()` from
`@onegrid/core`.

Theme and CSP security options control scoped runtime CSS variables without requiring
`unsafe-inline`:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  theme: {
    name: "finance",
    density: "standard",
    variables: {
      "--og-color-header-bg": "#eaf6f1"
    }
  },
  security: {
    csp: {
      nonce: window.__CSP_NONCE__
    }
  }
};
```

Set `security.csp.disableStyleInjection: true` when your application provides all OneGrid CSS
variables from static stylesheets. OneGrid does not use `eval`, `new Function`, or inline event
handler attributes.

`theme.name` maps to data-driven built-in presets from `@onegrid/themes/default.css`:
`default`, `clean`, `compact`, `dark`, and `high-contrast`. `theme.density` controls the row/header
scale independently with `comfortable`, `standard`, or `compact`.
For SI projects, `createSiTheme()` from `@onegrid/themes` converts organization design tokens into
the same `theme.variables` shape.

`security.html` controls the only code path that may write sanitized HTML:

```ts
import { createAllowlistHtmlSanitizer } from "@onegrid/dom";

const options: GridOptions<OrderRow> = {
  columns,
  data,
  security: {
    html: {
      allowHtmlRenderer: true,
      trustedTypesPolicyName: "onegrid-orders",
      sanitizer: createAllowlistHtmlSanitizer()
    },
    url: {
      allowedProtocols: ["https:", "mailto:", "tel:"]
    }
  }
};
```

Without both `allowHtmlRenderer: true` and a sanitizer, `kind: "html"` renderer output is displayed
as text. External sanitizer output is post-sanitized by OneGrid before it reaches the `innerHTML`
sink, so blocked tags, inline event attributes, inline styles, and unsafe URL protocols are still
removed if the adapter is misconfigured. Element renderer URL attributes are filtered through
`security.url.allowedProtocols`.

Menu options control row and cell context menu items while header menu availability remains under
`columnUi.menu`:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  columnUi: { menu: true, resize: true, autoSize: true },
  filtering: { enabled: true },
  clipboard: { enabled: true },
  contextMenu: {
    enabled: true,
    defaultItems: ["copyCell", "copyRow", "copyWithHeaders", "startEdit"],
    items: ({ row }) => [
      { id: "flag", label: `Flag ${row.id}`, scope: "row" }
    ]
  }
};
```

Context menus open from body cells through right click, `Shift+F10`, or the Context Menu key.
Custom items receive `ContextMenuContext<TData>` with row, row key, field, column, and value data.

Row model capabilities are intentionally explicit. Use the
[Row Model Capability Matrix](../features/row-model-capabilities) before choosing a model for
large data, server grouping, pivot, transactions, or lazy hierarchy. The same information is exported
from `@onegrid/core` as `rowModelCapabilityMatrix` and `getRowModelCapabilityProfile()`.

Infinite row model options are configured separately from `data` because rows are requested through
`DataSource.getRows` blocks:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  rowModel: "infinite",
  dataSource,
  infinite: {
    blockSize: 100,
    maxBlocksInCache: 10,
    initialRowCount: 1_000_000
  }
};
```

Server row model options keep browser execution separate from server execution. Sort, filter,
group, aggregate, and pivot models are sent to `DataSource.getRows`:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  rowKey: "id",
  rowModel: "server",
  dataSource,
  server: { pageSize: 100, initialPage: 0 },
  sorting: { serverOnly: true, model: sortModel },
  filtering: { serverOnly: true, model: filterModel },
  grouping: { serverOnly: true, model: groupModel },
  aggregation: { serverOnly: true, model: aggregateModel },
  pivot: { serverOnly: true, model: pivotModel }
};
```

Client pivot uses the same model but executes in `@onegrid/core` against the filtered client row
set:

```ts
const pivotModel = {
  rows: ["region", "agency"],
  columns: ["quarter"],
  values: [
    { field: "amount", function: "sum", alias: "amountTotal", label: "Amount" },
    { field: "budget", function: "avg", alias: "avgBudget", label: "Avg budget" }
  ],
  totals: "both",
  subtotals: true
};

const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowModel: "client",
  pivot: {
    enabled: true,
    panel: true,
    model: pivotModel
  }
};
```

Pagination options keep UI state one-based while server requests remain zero-based:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  rowKey: "id",
  rowModel: "server",
  dataSource,
  pagination: {
    mode: "server",
    position: "bottom",
    page: 1,
    pageSize: 100,
    pageSizeOptions: [50, 100, 250],
    pageGroupSize: 5
  }
};
```

For cursor pagination, set `pagination.mode: "cursor"` and optionally provide the first
`cursor`. `DataSource.getRows` receives `request.cursor`, and `nextCursor` from the result is
stored for the next page.

Viewport row model options describe the synchronized row window. The browser renders only the
current viewport entries while the data source owns the full row count:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  rowKey: "id",
  rowModel: "viewport",
  dataSource,
  viewport: {
    rowHeight: 30,
    viewportSize: 20,
    overscan: 2,
    prefetchRows: 40,
    maxCachedRanges: 4,
    initialRowCount: 10_000_000
  }
};
```

Tree row model options define the hierarchy contract and selection policy:

```ts
const options: GridOptions<TreeRow> = {
  columns,
  data,
  dataSource,
  rowKey: "id",
  rowModel: "tree",
  tree: {
    treeColumnField: "name",
    childrenField: "children",
    hasChildrenField: "hasChildren",
    indentSize: 22,
    expandedKeys: ["FIN"],
    filterPolicy: "withAncestors",
    sortPolicy: "siblings",
    selection: { policy: "descendants" }
  }
};
```

Use `tree.serverOnly: true` when `DataSource.getChildren()` owns child filtering and
sibling sorting. Lazy child requests receive the current `filtering.model` and
`sorting.model`.

Base layout options define renderer sizing and summary/footer layout without adding DOM
dependencies to core:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  rowModel: "client",
  layout: {
    width: "100%",
    height: 360,
    bodyHeight: 360
  },
  summary: {
    enabled: true,
    position: "bottom",
    sticky: true
  }
};
```

Pinned panes are driven by column definitions or `columnState`:

```ts
const columns: ColumnDef<OrderRow>[] = [
  { field: "id", pinned: "left" },
  { field: "amount", summary: "sum" },
  { field: "status", pinned: "right" }
];
```

Frozen row and column options provide the same behavior from `GridOptions` without changing each
column definition:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  frozenRows: { top: 2, bottom: 1 },
  frozenColumns: {
    left: ["id", "region"],
    right: ["status"]
  }
};
```

`frozenRows.top` and `frozenRows.bottom` are counts from the rendered row model. They become
separate non-scrolling rowgroups, while the center body keeps row virtualization and custom
scrollbar behavior. `frozenColumns.left/right` use column ids, which default to `field` when a
column does not define an explicit `id`.

Summary rows are driven by `ColumnDef.summary` and can be rendered at the top, bottom, or both
positions:

```ts
const options: GridOptions<OrderRow> = {
  columns: [
    { field: "id", summary: { kind: "count", label: "Rows" } },
    { field: "amount", summary: { kind: "sum", aggregateKey: "amountTotal", label: "Total" } },
    { field: "region", summary: { kind: "distinct-count", label: "Regions" } }
  ],
  data,
  rowKey: "id",
  summary: { enabled: true, position: "both", sticky: true },
  aggregation: {
    model: {
      fields: [{ field: "amount", function: "sum", alias: "amountTotal" }]
    }
  }
};
```

`SummaryDef.aggregateKey` maps a summary cell to `DataSource.getRows().aggregate.values` in server
row models. `SummaryDef.label` sets the visible label inside the summary cell. Client summaries are
calculated from the filtered row set.

Row virtualization options bound rendered DOM rows to viewport size plus overscan. `segmented`
scroll configuration is used by large logical row ranges and does not imply that client mode should
hold 10M to 100M rows in memory:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  rowModel: "client",
  rowHeight: 32,
  virtualization: {
    enabled: true,
    rowHeight: 32,
    overscan: { before: 3, after: 5 },
    maxDomRows: 64,
    segmented: true,
    maxScrollHeight: 24_000_000
  }
};
```

Segmented viewport mode for 10M to 100M logical rows uses a fixed numeric row height. Do not combine
that mode with `rowHeight: "auto"` or per-row height callbacks. Variable-height rows belong to local
or smaller virtualized datasets where measuring rendered rows does not require a full logical height
index.

Column virtualization is nested under `virtualization.columns` and applies to the center pane.
Pinned left and right columns remain outside the horizontal virtual window:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  rowKey: "id",
  rowModel: "client",
  virtualization: {
    enabled: true,
    columns: {
      enabled: true,
      overscan: { before: 1, after: 2 },
      maxDomColumns: 10
    }
  }
};
```

Cell merge layout is enabled through `merge`. Individual columns opt into value/custom merge, and
server row models can return `mergeMeta` from `DataSource.getRows`:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  rowKey: "id",
  rowModel: "server",
  dataSource,
  merge: {
    enabled: true,
    getSpan: ({ row, field }) =>
      field === "memo" && row.memoSpan ? { rowSpan: 1, colSpan: 2 } : undefined
  }
};
```

Accessibility options are shared across DOM, React, and Vue. They provide the grid label,
optional description, and live region politeness without requiring wrapper-specific APIs:

```ts
const options: GridOptions<OrderRow> = {
  columns,
  data,
  accessibility: {
    label: "Orders grid",
    description: "Keyboard navigable order status data.",
    liveRegion: "polite"
  }
};
```
