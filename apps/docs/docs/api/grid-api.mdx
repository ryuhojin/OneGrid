# GridApi

`GridApi<TData>` is the public method contract for future engine and renderer implementations.

Virtual navigation methods use the shared `ScrollAlign` contract:

```ts
api.scrollToRow(2_499, "start");
api.scrollToColumn("metric32", "nearest");
```

Column methods take `columnId` as the preferred identifier. Field names still resolve as a
compatibility fallback, but duplicate-field and generated columns should use explicit `columnId`
values:

```ts
api.hideColumn("workflow-owner");
api.pinColumn("order-id", "left");
api.autoSizeColumn("risk-score");
```

Column state methods persist and restore the same `ColumnUiState` shape used by
`GridOptions.columnState` and `GridStateSnapshot.columnState`:

```ts
const columnState = api.getColumnState();
const completeColumnState = api.getColumnState({ includeDefaults: true });

api.setColumnState({
  order: ["order-id", "customer", "amount", "status"],
  columns: {
    amount: { width: 160 },
    status: { pinned: "right" },
    internalNotes: { hidden: true }
  },
  groups: {
    workflow: { open: true }
  }
});

const result = api.applyColumnState({
  state: {
    order: ["amount", "order-id"],
    columns: {
      amount: { width: 180 },
      internalNotes: { hidden: true }
    }
  },
  defaultState: { pinned: null },
  applyOrder: true
});

if (!result.applied) {
  console.warn("Unknown column ids", result.missingColumnIds);
  console.warn("Unknown column group ids", result.missingGroupIds);
}

api.setColumnGroupOpen("workflow", true);
api.toggleColumnGroup("workflow");
api.resetColumnState();
```

`setColumnState()` replaces the current column UI state with the provided state and rerenders by
default. `applyColumnState()` is the safer incremental API for saved views and user preferences:
it applies only supplied column patches, can move only the supplied ordered columns, can apply a
`defaultState` to unspecified leaf columns, and returns missing column ids instead of silently
dropping them. Set `ignoreMissingColumns: true` only when restoring a state from an older schema.
`getColumnState({ includeDefaults: true })` returns a full leaf-column snapshot including effective
width, hidden, pinned values, and column group open states. `resetColumnState()` returns to
`GridOptions.columnState` plus any configured `frozenColumns` defaults.

Column state APIs honor `ColumnDef` policy flags. Locked visibility, pinned state, resize, and
position settings are ignored rather than partially applied, so saved views cannot hide required
columns, unpin locked columns, or move fixed-position columns out of their current slot.

Before-event methods attach cancellable handlers after constructor options are created:

```ts
const off = api.onBefore("beforeFilterChange", (event) => {
  if (event.event.filterModel.quickText === "restricted") {
    event.preventDefault("restricted quick filter");
  }
});

api.offBefore("beforeFilterChange", handler);
off();
```

`onBefore()` and `offBefore()` use `GridBeforeEventMap<TData>` names. Calling
`preventDefault()` prevents the pending sort, filter, page, selection, column state, or edit
mutation. Normal `events.*` handlers still run only after the change succeeds.

State methods capture and restore runtime UI state. `getState()` returns a `GridStateSnapshot`
containing column UI state, sort/filter/group models, selection, pagination, scroll offsets, and
locale. It also includes `rowModelState`, which captures row-model runtime state such as infinite
append position, server page/group/cursor state, viewport range metadata, and tree expansion or
selection state. `setState()` applies only the fields provided, so applications can restore a saved
view without remounting the grid:

```ts
const saved = api.getState();

api.setState({
  columnState: saved.columnState,
  sortModel: saved.sortModel,
  filterModel: saved.filterModel,
  rowModelState: saved.rowModelState,
  pagination: { page: 1, pageSize: 100 },
  scroll: { top: 0, left: 0 }
});
```

The same snapshot shape is accepted by `GridOptions.initialState` for first render. `setState()` is
not a data-loading API; it does not replace `columns`, `data`, `dataSource`, or renderer/security
configuration. Remote row model caches are not serialized; restored infinite, server, and viewport
models re-request their data through the configured `DataSource`.

Sorting methods update the shared `SortModel[]` used by header state, client row model sorting,
and remote row model requests:

```ts
api.setSortModel([{ columnId: "amount", field: "amount", direction: "desc" }]);
const sortModel = api.getSortModel();
```

Filtering methods update the shared `FilterModel` used by client evaluation and remote row model
requests:

```ts
api.setFilterModel({
  quickText: "office",
  conditions: [
    { columnId: "status", field: "status", kind: "set", operator: "in", value: ["Approved"] }
  ]
});

const filterModel = api.getFilterModel();
```

Grouping methods update the shared `GroupModel` used by client grouping and server group requests:

```ts
api.setGroupModel({ fields: ["region"], expandedKeys: ["group:region=Capital"] });
api.expandGroup("group:region=Regional");
api.collapseGroup("group:region=Capital");
api.toggleGroup("group:region=Digital");

const groupModel = api.getGroupModel();
```

For server row models, `expandGroup("Capital")` sets `groupKeys` to `["Capital"]` before the next
`DataSource.getRows()` call. `collapseGroup("Capital")` returns to the root group list.

Pivot methods update the shared `PivotModel` used by client pivot calculation and server pivot
requests:

```ts
api.setPivotModel({
  rows: ["region", "agency"],
  columns: ["quarter"],
  values: [{ field: "amount", function: "sum", alias: "amountTotal" }],
  totals: "both",
  subtotals: true
});

const pivotModel = api.getPivotModel();
```

Tree methods operate on the tree row model and reuse the configured lazy-loading and cascade
selection policy:

```ts
await api.expandTreeNode("DIG-AUD");
api.collapseTreeNode("CAP");
await api.toggleTreeNode("REG");
api.selectTreeNode("DIG", true);

const selectedTreeKeys = api.getTreeSelection();
```

`expandTreeNode()` and `toggleTreeNode()` return a promise because expanding a lazy node may call
`DataSource.getChildren()`.

Editing methods use the shared `CellPosition` contract. `field` remains the data key, while
`columnId` disambiguates duplicate-field columns. `startEdit()` opens the renderer editor for the
visible cell, and `stopEdit()` commits or cancels the active editor:

```ts
api.startEdit({ rowIndex: 0, rowKey: "ED-0001", columnId: "title", field: "title" });
api.stopEdit({ commit: true, validate: true });
api.stopEdit({ commit: false });
```

The DOM renderer also starts editing from Enter, F2, printable keys, and cell double click.
`stopEdit({ commit: true })` emits `cellEditCommitted` with `trigger: "api"`;
`stopEdit({ commit: false })` emits `cellEditCancelled` with `reason: "api"`.
When `editing.commitMode` is `"batch"`, `stopEdit({ commit: true })` stages the cell value instead
of finalizing it. Use the batch edit session API to group, inspect, finalize, or discard staged
history:

When `editing.readOnly` is `true`, `stopEdit({ commit: true })` emits `cellEditRequested` instead
of mutating or staging row data. Controlled applications should update their own store and then call
`setData()` or `updateRows()` with the accepted row.

```ts
const session = api.startBatchEditSession({
  id: "approval-draft",
  label: "Approval draft"
});

const activeSession = api.getBatchEditSession();
const pending = api.getPendingEdits();
await api.commitBatchEditSession();
api.cancelBatchEditSession();
```

Each pending edit includes `rowKey`, `position.field`, `position.rowIndex`, `sourceIndex` when the
renderer can resolve the original client data index, `previousValue`, `nextValue`, and `stagedAt`.
`startBatchEditSession()` is optional; if a grid in batch commit mode stages a value first, the DOM
renderer creates an implicit session. `commitPendingEdits()` and `cancelPendingEdits()` remain
supported aliases for existing integrations.

Undo and redo operate on accepted edit steps. Before batch commit they update the staged pending
history; after final commit they restore the committed row snapshot directly:

```ts
api.undoEdit();
api.redoEdit();

const history = api.getEditHistoryState();
api.clearEditHistory();
```

`getEditHistoryState()` exposes `canUndo`, `canRedo`, and the current stack sizes for toolbar state.
External data replacement through `setData()`, `appendRows()`, `updateRows()`, or `removeRows()`
clears the edit stack because row identity and stored snapshots may no longer match. For
client-backed data, these methods return a `ClientRowTransactionResult` so integrations can inspect
what changed:

```ts
const result = api.updateRows([
  { rowKey: "ED-0002", row: { budget: 430000 } },
  { rowKey: "ED-MISSING", row: { budget: 0 } }
]);

if (result) {
  console.log(result.updated);
  console.log(result.rejected);
}
```

The result includes `added`, `updated`, `removed`, `rejected`, `rowCountBefore`,
`rowCountAfter`, `acceptedRowCount`, and `rejectedRowCount`. Missing row keys and duplicate
transaction requests are reported through `rejected`. Server and remote row models may return
`undefined` because the mutation is handled asynchronously by the configured `DataSource`.

Selection methods expose the same core state used by pointer and keyboard interaction:

```ts
api.selectRows(["SEL-0001", "SEL-0002"]);
api.selectCell({ rowKey: "SEL-0001", rowIndex: 0, field: "agency", columnIndex: 2 });
api.selectCellRange(
  { rowKey: "SEL-0001", rowIndex: 0, field: "agency", columnIndex: 2 },
  { rowKey: "SEL-0002", rowIndex: 1, field: "memo", columnIndex: 4 }
);
api.selectAllVisibleRows();
api.selectServerDataset();

const selection = api.getSelectionState();
const rows = api.getSelectedRows();
```

`selectServerDataset()` produces a server-selection token instead of materializing remote rows.

Pagination methods use one-based page numbers and keep wrappers aligned with the DOM renderer:

```ts
api.setPage(3);
const page = api.getPage();

api.setPageSize(100);
const pageSize = api.getPageSize();
```

Changing page size resets to page 1. Server and cursor pagination reload the server row model with
the new `pageSize`; client pagination rerenders the paged visible row slice.

Clipboard methods use the current selection and the same edit validation pipeline as normal cell
editing:

```ts
await api.copyToClipboard({ selectedOnly: true, includeHeaders: true });
await api.pasteFromClipboard("Grant review\t410000");
```

`copyToClipboard()` writes `text/plain` only. `pasteFromClipboard()` parses TSV, skips read-only or
out-of-range targets, and emits normal edit commit or validation events for accepted cells.

Export and import methods use the shared core converters. Export does not execute renderer HTML, so
user data remains text unless the application explicitly consumes the returned file content:

```ts
const xlsx = await api.exportData({
  format: "xlsx",
  selectedOnly: true,
  includeHeaders: true,
  preserveVisualLayout: true
});

const imported = await api.importData(xlsx.content as Uint8Array, {
  format: "xlsx",
  mode: "replace",
  hasHeaders: true,
  parseRow: (record) => ({
    id: String(record.id),
    amount: Number(record.amount)
  })
});
```

`exportData()` returns `GridExportResult` with `content`, `mediaType`, and `filename`.
`importData()` returns imported rows plus rejected parser rows. The default import mode is
`"replace"`, so CSV/XLSX/JSON import clears existing client-side rows before applying imported rows.
Use `mode: "append"` only when the imported rows should be added after the current client rows.

Locale methods switch renderer chrome and formatter context values without remounting the grid:

```ts
api.setLocale("ko-KR");
const activeLocale = api.getLocale();
```

The DOM renderer rerenders footer, overlays, pagination, ARIA live messages, and formatted cell
values using the selected locale.

Theme updates use the same CSP-aware runtime style path as initial grid creation:

```ts
api.applyTheme({
  name: "dark",
  density: "compact",
  variables: {
    "--og-color-focus-ring": "#d7191f"
  }
});
```

When `security.csp.nonce` is configured, the inserted runtime style tag receives that nonce. When
`security.csp.disableStyleInjection` is true, `applyTheme()` updates the theme metadata without
inserting a style tag. Built-in theme names and density values are documented in
`features/theme-foundation`; SI token mapping is documented in `features/si-customization`.

Plugin methods expose lifecycle-bound extension metadata without wrapper-specific behavior:

```ts
if (api.hasPlugin("audit")) {
  const menuItems = api.getPluginExtensions<{ label: string }>("menu.context");
}
```

`getPluginExtensions()` returns ordered, immutable extension records registered by plugins.
Extension payloads are metadata or adapter hooks; they do not bypass renderer security or sanitizer
policy.
