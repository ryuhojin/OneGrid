# ColumnDef

`ColumnDef<TData>` supports data columns and grouped columns. Group headers and merge-aware layout
are first-class design constraints even though the initial renderer shell only paints leaf columns.

Use `columnId` as the stable UI/API identity for a column. Keep `field` as the optional row data
binding key. This matters when multiple columns read the same field, a column is generated from
`valueGetter`, or state must survive reorder, pinning, wrapper slots, and header merge rules.

```ts
const columns: ColumnDef<OrderRow>[] = [
  { columnId: "order-id", field: "id", headerName: "ID" },
  { columnId: "risk-score", field: "score", headerName: "Risk" },
  {
    columnId: "risk-band",
    headerName: "Risk Band",
    valueGetter: ({ row }) => row.score >= 80 ? "high" : "normal"
  },
  {
    columnId: "row-actions",
    headerName: "Actions",
    renderer: { kind: "text", render: () => "Open" }
  },
  {
    columnId: "workflow",
    headerName: "Workflow",
    children: [
      { columnId: "workflow-status", field: "status" },
      { columnId: "workflow-owner", field: "owner" }
    ]
  }
];
```

When `field` is omitted, provide `columnId` for stable state, events, sort/filter model references,
renderer slots, and export/import mapping. Fieldless columns should use `valueGetter`, `renderer`,
or `valueSetter` when they need data access or editing behavior.

Legacy `id` and `groupId` remain supported as compatibility aliases, but new code should prefer
`columnId` for column state, menu actions, wrapper renderer slots, and `GridApi` column methods.

Use `GridOptions.defaultColumnDef` and `GridOptions.columnTypes` when many columns share the same
sizing, filter, editor, formatter, or renderer behavior:

```ts
const columns: ColumnDef<OrderRow>[] = [
  { columnId: "order-id", field: "id", headerName: "ID" },
  { field: "amount", headerName: "Amount", type: "money", width: 180 },
  { field: "status", headerName: "Status", type: ["text", "statusPill"] }
];

const options: GridOptions<OrderRow> = {
  columns,
  defaultColumnDef: { minWidth: 96, resizable: true, sortable: true },
  columnTypes: {
    money: { type: "number", filter: "number", editor: "number", width: 140 },
    statusPill: { renderer: statusRenderer }
  }
};
```

Defaults are resolved in a stable order: `defaultColumnDef`, defaults referenced by
`defaultColumnDef.type`, defaults referenced by `ColumnDef.type`, and finally explicit column
properties. Explicit column properties always win.

Columns can opt into body cell merge calculation:

```ts
const columns: ColumnDef<OrderRow>[] = [
  { field: "region", merge: { mode: "value" } },
  {
    field: "memo",
    merge: {
      mode: "custom",
      colSpan: ({ row }) => row.memoSpan ? 2 : 1
    }
  }
];
```

String cell values are still rendered as text by default. Merge only changes layout and ARIA span
metadata; it does not opt into HTML rendering.

Columns participate in sorting unless `sortable: false` is set. A column can define an initial sort
direction or a custom comparator:

```ts
const columns: ColumnDef<OrderRow>[] = [
  { field: "amount", type: "number", sort: "desc" },
  {
    field: "status",
    sortComparator: (left, right) => statusRank[String(left)] - statusRank[String(right)]
  }
];
```

`sortComparator` receives the two resolved values plus the left and right row context. When a
`valueGetter` is present, client sorting uses the getter value instead of reading `row[field]`.

Columns can opt into text, number, date, boolean, set, or custom filtering:

```ts
const columns: ColumnDef<OrderRow>[] = [
  { field: "agency", filter: "text" },
  { field: "amount", type: "number", filter: "number" },
  { field: "status", filter: { kind: "set", values: ["Approved", "Review"] } },
  {
    field: "service",
    filter: {
      kind: "custom",
      predicate: ({ value, filterValue }) =>
        String(value).toLocaleLowerCase().includes(String(filterValue).toLocaleLowerCase())
    }
  }
];
```

`FilterOptions.serverOnly` keeps a column filter in the remote request model while skipping
client evaluation. `valueGetter` is used by client filters and quick filter when present.

Columns opt into editing with `editable` and `editor`. The core lifecycle resolves the editor,
parses raw values, runs sync or async validators, and applies either `valueSetter` or a shallow
field update:

```ts
const columns: ColumnDef<OrderRow>[] = [
  {
    field: "amount",
    type: "number",
    editable: true,
    editor: "number",
    validator: (value) => Number(value) < 0 ? ["Amount must be positive"] : []
  },
  {
    field: "status",
    editable: true,
    editTrigger: "singleClick",
    editor: {
      kind: "select",
      options: [
        { value: "Draft", label: "Draft" },
        { value: "Approved", label: "Approved" }
      ]
    }
  },
  {
    field: "code",
    editable: true,
    editor: {
      kind: "custom",
      validate: async (value) => String(value).startsWith("ERR")
        ? ["Code cannot start with ERR"]
        : []
    },
    parser: (value) => String(value).trim().toUpperCase()
  }
];
```

Supported editor kinds are `text`, `number`, `date`, `datetime`, `checkbox`, `select`,
`multi-select`, `radio`, `textarea`, `autocomplete`, and `custom`.

`editTrigger` overrides `GridOptions.editing.startMode` for a single column. Supported values are
`doubleClick`, `singleClick`, and `manual`. `manual` suppresses pointer-started editing for that
column while keeping Enter, F2, printable-key, and `GridApi.startEdit()` entry points available.
Checkbox editor columns default to `singleClick` when neither the grid nor column supplies a policy.

Cell renderers are secure by default. Omitted renderers and `kind: "text"` renderers write strings
with `textContent`. `kind: "element"` uses DOM node creation and filters unsafe tags, inline event
attributes, inline style attributes, and unsafe URL protocols. `kind: "html"` is interpreted only
when `GridOptions.security.html.allowHtmlRenderer` is true and a sanitizer is supplied:

```ts
const columns: ColumnDef<OrderRow>[] = [
  {
    field: "status",
    renderer: {
      kind: "html",
      sanitize: true,
      render: ({ value }) => `<strong>${String(value)}</strong>`
    }
  }
];
```
